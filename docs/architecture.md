# Архітектура проєкту Library System

## Домен проєкту

### Яку проблему вирішує система?
Система автоматизує роботу бібліотеки: дозволяє користувачам шукати книги, бронювати їх онлайн, переглядати історію видач, а адміністраторам — керувати каталогом та обліком.

### Користувачі (ролі)
- **Гість (Guest)** — неавторизований користувач, може переглядати каталог книг
- **Користувач (User)** — зареєстрований користувач, може брати/повертати книги, переглядати історію, залишати відгуки
- **Адміністратор (Admin)** — керує книгами (CRUD), користувачами, видачею/поверненням

### Основні операції системи
1. Реєстрація та авторизація користувача
2. Пошук книг за назвою/автором (з пагінацією)
3. Взяття книги (бронювання) користувачем
4. Повернення книги
5. Перегляд історії позичених книг
6. Додавання/редагування/видалення книги (адміністратор)
7. Залишення відгуку про книгу

### Зовнішні системи
| Система | Призначення |
|---------|-------------|
| PostgreSQL | Основна база даних (книги, користувачі, видачі, відгуки) |
| SMTP-сервер | Відправлення email-сповіщень (підтвердження реєстрації, нагадування про повернення) |

## Архітектурний стиль

### Таблиця вибору архітектури

| Критерій | Аналіз для Library System |
|----------|---------------------------|
| Розмір команди | 1 студент (навчальний проєкт) |
| Складність бізнес-логіки | Середня: CRUD + правила видачі книг |
| Вимоги до масштабування | Низькі — навчальний проєкт |
| Терміни | MVP за 2-3 тижні |
| **Вибір** | **Layered Architecture (3-шарова) + MVC** |

### Обґрунтування вибору
- Простота реалізації та розуміння
- Чіткий розподіл обов'язків
- Легко тестувати кожен шар окремо
- Достатньо для навчального проєкту
